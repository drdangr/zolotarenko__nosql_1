# NoSQL HW1 (Theme 5) — Аналітична платформа для музичного стрімінгового сервісу

Документоорієнтована модель даних у MongoDB на базі датасету **Spotify Tracks Dataset**
(~114 000 треків). Робота охоплює проєктування схеми, запити, аналітичні пайплайни
та оптимізацію через індекси.

## 1. Налаштування оточення

### 1.1. База даних

У цій роботі замість MongoDB Atlas використано **локальний MongoDB у Docker**.
Для частин 1–4 (проєкція схеми, запити, агрегації, `explain()` / індекси /

## 2. Схема даних (колекція `tracks`)

Сирий CSV — плоска таблиця. Через Aggregation Pipeline (`scripts/02_transform.js`)
дані приведено до документоорієнтованої форми. Приклад документа:

```json
{
  "_id": { "$oid": "6a0a181cbf5e4209a4b98526" },
  "track_id": "5SuOikwiRyPMVoIQDJUgSV",
  "track_name": "Comedy",
  "album_name": "Comedy",
  "explicit": false,
  "popularity": 73,
  "duration_ms": 230666,
  "track_genre": "acoustic",
  "artists": ["Gen Hoshino"],
  "audio_features": {
    "danceability": 0.676,
    "energy": 0.461,
    "loudness": -6.746,
    "speechiness": 0.143,
    "acousticness": 0.0322,
    "instrumentalness": 0.00000101,
    "liveness": 0.358,
    "valence": 0.715,
    "tempo": 87.917,
    "key": 1,
    "mode": 0,
    "time_signature": 4
  },
  "duration_sec": 230.7,
  "popularity_tier": "high"
}
```

Ключові рішення:

- `artists` — **масив** (рядок розбито по `;` з обрізанням пробілів).
- `audio_features` — **вкладений об'єкт** з усіма аудіофічами.
- `duration_sec` — `duration_ms / 1000`, округлено до 1 знака.
- `popularity_tier` — `high` (≥70) / `medium` (40–69) / `low` (<40).
- Службовий стовпець-індекс CSV (`Unnamed: 0`) та `artists_raw` у фінальну
  схему не потрапляють.

Кількість документів: **113 999** (з 114 000 рядків відкинуто 1 запис без
виконавця/назви треку — фільтр у `01_load_data.py`).

Розподіл за `popularity_tier`: `low` — 65 412, `medium` — 43 115, `high` — 5 472.

## 3. Відповіді на теоретичні питання

### Частина 1

**1. Чому аудіо-характеристики винесені в окремий об'єкт `audio_features`, а не зберігаються плоско?**

Вкладення групує концептуально однорідні дані: усі акустичні метрики
(`danceability`, `energy`, `tempo`, …) логічно належать одній сутності
«звукова характеристика треку». Переваги:

- *Семантична чистота*: метадані треку (`track_name`, `album_name`,
  `popularity`) відокремлені від сигнальних фіч. Документ читається як
  модель предметної області, а не як «звалище колонок».
- *Зручність роботи цілим блоком*: легко спроєктувати/приховати весь
  набір фіч однією дією (`audio_features: 0`), передати об'єкт у код як
  єдину структуру.
- *Менший ризик колізій імен* при подальшому розширенні схеми.

Коли вкладення **створює проблеми**: якщо до окремих фіч постійно йдуть
вибіркові запити з індексами — шлях стає довшим (`audio_features.danceability`),
а індекси по вкладених полях треба створювати явно. Якщо вкладений об'єкт
росте безмежно або перетворюється на масив об'єктів змінної довжини —
документ роздувається (ліміт 16 МБ) і запити ускладнюються. У нашому випадку
набір фіч фіксований і невеликий — вкладення виправдане.

**2. Чому виконавці зберігаються як масив, а не як рядок?**

У треку може бути кілька виконавців (фіти, дуети). Рядок `"A;B"` — це
денормалізований текст, який доводиться парсити в кожному запиті. Масив
`["A", "B"]` дає природні запити:

- пошук усіх треків артиста: `{ artists: "Rema" }` (працює без `$regex`);
- розгортання для агрегації по артистах: `$unwind: "$artists"`
  (використано у Частині 2 завд. 2 та Частині 3 завд. 1);
- оператори масивів: `$size`, `$in`, `$all`, `"artists.1": {$exists:true}`
  (пошук колаборацій — ≥2 виконавці).

З рядком усі ці запити або неможливі, або повільні/крихкі через регулярні вирази.

**3. Що таке `$out` і чим він відрізняється від `$merge`?**

Обидва — термінальні стадії, що пишуть результат пайплайна в колекцію.

- `$out` **повністю замінює** цільову колекцію (атомарно перестворює її).
  Простий, але «все або нічого»: будь-який наявний вміст і несумісні
  індекси втрачаються. Підходить для повної перебудови похідної колекції —
  саме наш кейс (`tracks_raw` → `tracks`).
- `$merge` (з MongoDB 4.2) — гнучкий upsert: може оновлювати/доповнювати
  існуючі документи за ключем (`whenMatched`: replace/merge/keepExisting/
  fail/pipeline; `whenNotMatched`: insert/discard/fail), писати в іншу БД,
  не руйнує колекцію цілком. Підходить для інкрементальних оновлень
  (матеріалізовані в'ю, дозавантаження порціями).

Правило: повна одноразова трансформація → `$out`; інкрементальне
оновлення наявної колекції → `$merge`.

### Частина 2

**1. Для чого використовується `$unwind`?**

`$unwind` розгортає масив: з одного документа з масивом із N елементів
робить N документів, у кожному — один елемент масиву на місці поля. Це
дозволяє групувати/рахувати **по елементах масиву**. У завданні 2 без
`$unwind: "$artists"` ми б групували по комбінаціях співавторів, а не по
кожному окремому виконавцю — і статистика «кількість треків артиста» була б
неправильною.

**2. Чим `$stdDevPop` відрізняється від `$stdDevSamp`?**

Обидва рахують стандартне відхилення, але з різним знаменником:

- `$stdDevPop` — **популяційне** (ділення на `N`). Використовується, коли
  наявні дані і є вся генеральна сукупність.
- `$stdDevSamp` — **вибіркове** (ділення на `N − 1`, поправка Бесселя).
  Використовується, коли дані — лише вибірка, і ми оцінюємо відхилення
  всієї популяції.

У завданні 3 ми трактуємо всі треки жанру як повну сукупність цього жанру
в датасеті, тому коректно `$stdDevPop` (як і вимагає умова). На великих
жанрах (по 1000 треків) різниця між Pop і Samp практично нульова, але
методологічно вибір знаменника має значення на малих вибірках.

### Частина 3

**1. Завдання 1: поріг кількості треків артиста.**

Фільтр `tracks_count >= 5` відсікає артистів з малою кількістю треків.

- *Поріг = 1*: у вибірку потраплять усі артисти, зокрема ті, хто має
  1–2 треки. Серед них з'являться «одноразові» виконавці з одним дуже
  популярним синглом і середньою популярністю ~90–100 — вони витіснять
  стабільних артистів з топу. Середнє по 1 значенню — це не «середня
  популярність артиста», а популярність єдиного треку: статистично
  ненадійно, рейтинг стає шумним.
- *Поріг > 50*: залишаться лише дуже «плідні» виконавці. У цьому датасеті
  таких майже немає (більшість артистів мають одиниці–десятки треків),
  тому результат сильно скоротиться або стане порожнім; те, що лишиться,
  матиме «розмиту» середню популярність (багато треків різної якості),
  тож топ за середнім зміститься вниз за значеннями.

Висновок: поріг — це компроміс між статистичною надійністю середнього
(більше треків = стабільніша оцінка) і покриттям (не відсікти всіх).

**2. Завдання 3: поріг 100 vs 50 треків на жанр.**

**Результат не зміниться.** Spotify Tracks Dataset збалансований штучно:
кожен із 114 жанрів містить рівно **1000 треків** (перевірено
агрегацією `$group` за `track_genre`). Тому будь-який поріг у діапазоні
1…1000 (зокрема і 100, і 50) пропускає геть усі жанри — фільтр
`tracks_count >= 100` не відсіює нічого. Поріг почав би впливати лише
якби деякі жанри мали < 100 (або < 50) треків, чого в цьому датасеті
немає. Це гарний приклад, чому корисно спершу подивитися на розподіл
даних, а не сліпо довіряти «захисному» фільтру.

## 4. Частина 4 — Аналіз індексів (`explain()` до/після)

Метрики з `explain("executionStats")` (локальний MongoDB 7, том із даними).

### Завдання 1 — точний збіг + діапазон + сортування

Запит:

```js
db.tracks.find({ track_genre: "pop",
                  "audio_features.danceability": { $gte: 0.7 } })
         .sort({ popularity: -1 })
```

| Метрика              | ДО індексу        | ПІСЛЯ індексу                 |
|----------------------|-------------------|-------------------------------|
| Стадії плану         | `SORT ← COLLSCAN` | `FETCH ← IXSCAN`              |
| nReturned            | 354               | 354                           |
| totalKeysExamined    | 0                 | 412                           |
| totalDocsExamined    | **113 999**       | **354**                       |
| executionTimeMillis  | ~26               | ~1                            |

Створений індекс (правило **ESR**: Equality → Sort → Range):

```js
db.tracks.createIndex({ track_genre: 1, popularity: -1,
                        "audio_features.danceability": 1 })
```

**Що змінилося і чому:**

- *До*: `COLLSCAN` — повний перебір усіх 113 999 документів, потім окрема
  блокуюча стадія `SORT` у пам'яті (сортування результату вручну).
- *Після*: `IXSCAN` по складеному індексу. `track_genre` (рівність)
  одразу позиціонує діапазон ключів; `popularity: -1` у самому індексі
  дає **готовий порядок сортування** — стадія `SORT` зникла повністю;
  `danceability` як останнє поле звужує діапазон. `totalDocsExamined`
  впав з 113 999 до 354 (≈у 322 рази), час — з ~26 мс до ~1 мс.

**Як зрозуміти, що індекс використовується:**

- `winningPlan` містить стадію `IXSCAN` з `indexName: "idx_part4_task1"`
  (замість `COLLSCAN`);
- немає окремої стадії `SORT` (порядок забезпечує індекс);
- `totalKeysExamined > 0`, а `totalDocsExamined` близьке до `nReturned`
  (читаються майже лише потрібні документи, а не вся колекція).

### Завдання 2 — індекс для «музики для роботи»

```js
db.tracks.createIndex({ explicit: 1,
                        "audio_features.instrumentalness": 1,
                        "audio_features.speechiness": 1 })
```

Запит `{ explicit:false, instrumentalness:{$gt:0.5}, speechiness:{$lt:0.1} }`:

| Метрика             | Значення                       |
|---------------------|--------------------------------|
| Стадії плану        | `FETCH ← IXSCAN (idx_part4_task2)` |
| nReturned           | 16 141                         |
| totalKeysExamined   | 16 602                         |
| totalDocsExamined   | 16 141                         |

Порядок полів обрано за ESR: `explicit` — рівність (іде першим, максимально
звужує діапазон ключів), далі діапазонні `instrumentalness` / `speechiness`.
`IXSCAN` з `indexName: "idx_part4_task2"` у плані підтверджує використання
індексу; `totalDocsExamined == nReturned` — зайвих документів не читаємо.

### Завдання 3 — Покривний запит (covered query)

Запит: `db.tracks.find({ track_genre: "pop", popularity: { $gte: 70 } })`,
індекс із Завдання 1 існує
(`{ track_genre:1, popularity:-1, "audio_features.danceability":1 }`).

| Варіант                                          | Стадії плану                 | totalDocsExamined |
|--------------------------------------------------|------------------------------|-------------------|
| Як у завданні (без проєкції)                     | `FETCH ← IXSCAN`             | **317**           |
| З проєкцією `{_id:0, track_genre:1, popularity:1}`| `PROJECTION_COVERED ← IXSCAN`| **0**             |

**Відповідь: запит у тому вигляді, як він даний, — НЕ покривний.**

Covered query — це запит, який повністю обслуговується індексом: усі поля
з фільтра **і** з проєкції присутні в індексі, а сама колекція документів
не читається (`totalDocsExamined == 0`, стадія `PROJECTION_COVERED`,
відсутній `FETCH`).

Чому даний запит не покривний:

1. **Немає проєкції** → повертаються всі поля документа (`track_name`,
   `audio_features`, `artists`, …). Цих полів в індексі немає, тож MongoDB
   змушена робити `FETCH` — діставати повні документи з колекції
   (`totalDocsExamined = 317`).
2. **Повертається `_id`** (за замовчуванням), а `_id` не входить до
   складеного індексу `idx_part4_task1` — це саме по собі унеможливлює
   покриття навіть для вузької проєкції без явного `_id: 0`.

Що довелося б зробити, щоб запит став покривним: обмежити проєкцію
**виключно полями індексу** і явно прибрати `_id`, напр.
`find({track_genre:"pop", popularity:{$gte:70}}, {_id:0, track_genre:1, popularity:1})`.
Експеримент це підтверджує: з такою проєкцією план стає
`PROJECTION_COVERED ← IXSCAN`, `totalDocsExamined = 0` — індекс
самодостатній, колекція не читається взагалі.

## 6. Особисті спостереження

- **Дублікати по жанрах.** Датасет «штучно збалансований»: 114 жанрів ×
  1000 треків. Один і той самий трек присутній у кількох жанрах
  (напр. `LOKERA`, `Party` трапляються кілька разів з різним `track_genre`).
  Це означає, що метрики «по артистах» (Частина 2 завд. 2, Частина 3
  завд. 1) рахують одну пісню кілька разів. Я свідомо лишив схему «один
  рядок CSV = один документ із своїм `track_genre`», бо так вимагає
  завдання (поле `track_genre` у документі). Для «чистої» аналітики по
  унікальних треках варто було б додавати дедуплікацію за `track_id`.
- **ESR на практиці.** Найнаочніший ефект індексу — не стільки зменшення
  `docsExamined`, скільки зникнення блокуючої стадії `SORT`: правильний
  порядок полів в індексі віддає вже відсортований потік.
- **Covered query крихкий.** Достатньо забути `_id: 0` — і запит перестає
  бути покривним, навіть якщо всі інші поля в індексі. `explain()` —
  єдиний надійний спосіб це перевірити.
